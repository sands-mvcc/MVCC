Understanding MFA

Multifactor Authentication is a cornerstone of modern identity security. This lesson covers how it works, why it matters, and how to implement it effectively.

WHAT IS MFA?
Multifactor Authentication (MFA) is a security system that requires two or more distinct authentication factors from different categories to verify a user's identity before granting access to a resource. Its primary purpose is to add layered defense — even if one credential is compromised, an attacker must overcome additional, independent obstacles.
THE AAA FRAMEWORK
🔐
Authentication
WHO are you?
Verifying identity before granting access. MFA lives here. This is the first gate.
🎫
Authorization
WHAT can you do?
Determines permissions and access rights after identity is confirmed.
📋
Accounting
WHAT did you do?
Logs and tracks user activity for audit trails and forensics.
⚡ Why passwords alone aren't enough: In 2023, over 80% of data breaches involved compromised credentials. A stolen password provides zero resistance if no second factor is required. MFA is the single most effective control against credential-based attacks.

The 3 Core Factors

True MFA must combine factors from at least two distinct categories. Click each factor to explore its examples, strengths, and vulnerabilities.

🧠
FACTOR 01 — KNOWLEDGE
Something You KNOW
▼
Examples
  • Passwords and passphrases (e.g., "MyD0g$Name!sRex")
  • Personal Identification Numbers (PINs)
  • Security question answers (e.g., "Mother's maiden name")
  • Pattern locks (on mobile devices)
Strengths
Easy to deploy No hardware needed Familiar to users
Vulnerabilities
PHISHING BRUTE FORCE CREDENTIAL STUFFING KEYLOGGERS SHOULDER SURFING DATA BREACHES
Real-World Context: The 2024 RockYou2024 data dump contained 10 billion plaintext passwords. Knowledge-only authentication is effectively broken at scale — pairing with a second factor is essential.
📱
FACTOR 02 — POSSESSION
Something You HAVE
▼
Examples
  • Hardware tokens (e.g., YubiKey, RSA SecurID)
  • Authenticator apps generating TOTP (Google Authenticator, Authy)
  • Smart cards with embedded certificates (CAC, PIV)
  • Mobile device receiving SMS or Push notifications
  • One-Time Password (OTP) via email
Strengths
Hard to replicate remotely Requires physical access Out-of-band delivery
Vulnerabilities
PHYSICAL THEFT SIM SWAPPING (SMS) MFA FATIGUE REAL-TIME PHISHING DEVICE LOSS
⚡ MFA Fatigue Attack: Attackers spam an authenticated user with push notification requests, hoping they'll approve out of frustration or confusion. Mitigate with number-matching in push apps.
👁️
FACTOR 03 — INHERENCE
Something You ARE
▼
Examples
  • Fingerprint scan (capacitive or optical sensors)
  • Facial recognition (2D photo or 3D depth map)
  • Iris / retina scan (high-assurance environments)
  • Voice authentication (voiceprint analysis)
  • Palm vein recognition
Strengths
Extremely convenient Always with the user Hard to steal without presence
Vulnerabilities
IRREVERSIBLE IF COMPROMISED SPOOFING (DEEPFAKE/REPLICA) BIOMETRIC DB BREACH PRIVACY CONCERNS
⚡ Critical Difference: If your password is stolen, you change it. If your fingerprint data is stolen, you cannot change your fingerprint. This irreversibility is the most serious inherence vulnerability.
MODULE 03

Advanced & Emerging Factors

Modern authentication systems use passive, contextual signals to make continuous risk-based decisions beyond the three core factors.

📍
Location Factor
Somewhere You Are

Verifies access based on the user's physical or network location.

Examples:
  • GPS coordinates from mobile device
  • Corporate office IP address range
  • Geo-velocity detection (impossible travel)
  • Country-based allow/deny policies
Geo-velocity example: A user logs in from Chicago at 8:00 AM, then attempts login from London at 8:30 AM — physically impossible, so the system flags it as suspicious.
🎭
Behavioral Factor
Something You Do

Analyzes the unique way a user interacts with their device — continuously, not just at login.

Examples:
  • Keystroke dynamics (typing rhythm & speed)
  • Mouse movement patterns
  • Touchscreen swipe pressure & angle
  • Walking gait (wearable sensors)
Continuous Auth: Behavioral factors enable continuous authentication — instead of verifying identity once at login, the system constantly confirms the right person is still at the keyboard.
ADAPTIVE / RISK-BASED MFA
Adaptive MFA Engine
Risk-Based Authentication dynamically adjusts the number and type of factors required based on the perceived risk of each login attempt.

When risk is LOW (known device, normal location, typical hours) → fewer factors required.

When risk is HIGH (new device, foreign IP, unusual time, velocity anomaly) → additional factors triggered.

MODULE 04

Technical Mechanisms

Understanding the delivery and verification mechanics of each MFA method is essential for secure implementation decisions.

MECHANISM HOW IT WORKS FACTOR TYPE SECURITY LEVEL
OTP
 A temporary code valid for one login session only. Generated server-side and delivered via SMS, email, or voice call. POSSESSION MODERATE
TOTP
 Time-Based OTP using HMAC algorithm. Code refreshes every 30–60 seconds. Both server and app share a secret key. Offline capable. POSSESSION HIGH
PUSH NOTIFY
 An out-of-band message sent to a trusted mobile app asking the user to approve a login. User taps Approve or Deny. Simple but vulnerable to fatigue attacks. POSSESSION MODERATE
FIDO2 / WebAuthn
 Open standard using public-key cryptography. Private key never leaves the device. Biometric or PIN unlocks the local key. Phishing-resistant by design — server never sees a password. INHERENCE + POSSESSION VERY HIGH
SMS OTP
 Code delivered via text message to the user's phone number. Convenient but subject to SIM swapping, SS7 protocol attacks, and interception. POSSESSION LOW–MOD
SMART CARD / PKI
 Physical card with embedded chip containing X.509 digital certificate. User inserts card and enters PIN. Government and enterprise use (CAC, PIV). KNOWLEDGE + POSSESSION VERY HIGH
THE TOTP ALGORITHM (HOW IT WORKS)
TOTP Process Flow
01. Server & app share a secret key (provisioned via QR code)
02. Both compute: HOTP(key, floor(time/30))
03. Result is a 6-digit code
04. Code valid for 30–60 seconds
05. User enters code → server validates
FIDO2 / Passkey Flow
01. Server sends challenge to browser
02. Device prompts for biometric/PIN
03. Private key signs the challenge
04. Server verifies with public key
05. No password ever transmitted ✓
MODULE 05

MFA vs Two-Step Verification

These terms are often used interchangeably — incorrectly. The distinction is critical in security design.

✅ True MFA
Uses factors from two or more DIFFERENT categories.

Example combinations:
  • Password (Knowledge) + Fingerprint (Inherence)
  • PIN (Knowledge) + YubiKey (Possession)
  • Password + Authenticator App + Face ID
STRONGER ASSURANCE CROSS-CATEGORY
⚠️ Two-Step Verification
Uses two steps from the SAME category.

Examples (NOT true MFA):
  • Password + Security Question (both Knowledge)
  • Password + Recovery Email Password (both Knowledge)
If an attacker knows your password, they may also know or crack security questions from the same data source.

Key Terms

Critical vocabulary for MFA design, implementation, and security assessment.

AAA
Authentication, Authorization, and Accounting. The three pillars of access control. MFA operates within the Authentication phase, verifying identity before any authorization decisions are made.
CREDENTIAL STUFFING
An automated attack where leaked username/password pairs are tested against multiple services, exploiting password reuse. MFA is highly effective at blocking this attack even when credentials are known.
SIM SWAPPING
An attack where the adversary tricks a telecom provider into transferring the victim's phone number to a new SIM card under attacker control, enabling interception of SMS OTP codes.
OUT-OF-BAND (OOB)
A method of transmitting authentication data using a separate communication channel from the primary login attempt. Example: logging in on a laptop while receiving approval push on a smartphone.
ADAPTIVE MFA
Also called Risk-Based Authentication. Dynamically adjusts authentication requirements based on the assessed risk of each login attempt — evaluating device, location, time, and behavior signals.
TOTP
Time-Based One-Time Password. A dynamic 6-digit code computed using a shared secret key and the current time (HMAC algorithm). Refreshes every 30–60 seconds. Used by Google Authenticator, Authy, Microsoft Authenticator.
FIDO2 / WebAuthn
An open standard using public-key cryptography for passwordless and phishing-resistant MFA. The private key never leaves the device. Authentication is performed locally with a biometric or PIN.
MFA FATIGUE
A social engineering attack where an attacker repeatedly sends push notification MFA requests to an authenticated user, hoping they will accidentally or frustratedly approve one. Countered by number-matching and context-aware push apps.
GEO-VELOCITY
A risk signal that detects logins from locations that are geographically impossible to reach given the time elapsed since the last login — indicating a compromised account or unauthorized access.
